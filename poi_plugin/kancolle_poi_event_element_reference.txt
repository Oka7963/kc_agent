KanColle / poi battle-event element reference
Generated for Codex-readable use.

Scope:
- Based on the user's poi battle-event log fields discussed in this conversation.
- Extended with known fixed/macro-like values from public KanColle API notes and poi plugin API docs.
- Values marked "known/common" are not guaranteed to be exhaustive for future event maps or new mechanics.
- For final battle result, prefer poi @@BattleResult over raw battleresult response.

Legend:
- "observed" = seen in the supplied test log / plugin payload shape discussed here.
- "known/common" = public API/community reference value.
- "variable" = not a fixed enum; treat as data.
- "unknown/reserved" = value exists in logs/community notes but exact semantics are not stable or not verified.

Receiver unified JSONL format

The plugin payload remains unchanged over HTTP. `battle_receiver.py` converts
each accepted payload into one receiver-owned JSONL record. Only fields for
which the current packet contains usable data appear under `updates`.

| element | data type | description | possible value |
|---|---|---|---|
| schema | string | Receiver output schema identifier. | battle_receiver.v1 |
| packet_no | integer | Receiver-local monotonically increasing packet number for debugging. | 1, 2, 3, ... |
| error | object or null | Packet/API/ordering error. Null means the packet was structurally processed successfully. | null or `{code,message}` |
| accepted | boolean | Whether the plugin event was accepted into delivery state. | true, false |
| received_at | string | Receiver timestamp with local timezone. | ISO 8601 |
| event_id | string or null | Unchanged plugin event ID. | session UUID + plugin seq |
| session_id | string or null | Unchanged plugin session ID. | UUID-like string |
| seq | integer or null | Unchanged plugin FIFO sequence number. | 1, 2, 3, ... |
| event | string | Unchanged plugin event classification, or receiver_error/plugin_loaded for receiver records. | variable |
| phase | string | Plugin phase or receiver. | response, derived, poi_battle_result, receiver |
| source | string | Plugin source or battle_receiver. | game.response, game.response+poi-lib-battle, @@BattleResult, battle_receiver |
| path | string or null | Original API/receiver path. | variable |
| exported_at | integer or null | Unchanged plugin export timestamp. | Unix milliseconds |
| updates | object | Partial normalized data supplied by this packet. Missing/empty fields are omitted. | request, api, map, formation, night_battle, player_fleet, enemy_fleet, enemy_preview, battle, battle_result |
| raw_event | object, optional | Unchanged plugin payload, included only with receiver `--full-body`. | object |

Receiver `error.code` values currently include `missing_event`, `api_error`,
`simulation_invalid`, `duplicate`, `out_of_order`, `unsupported_path`,
`invalid_content_length`, `invalid_json_type`, `bad_json`, and
`processing_error`, `log_write_failed`.

Enemy HP arrays can contain non-numeric placeholders such as `"N/A"` for a
special enemy unit. The receiver preserves that unit and displays
`hp: "N/A/N/A"` with `damage: "unknown"`; it must not treat the placeholder as
zero or discard the ship.

Implementation notes for Codex:
1. Do not assume response_body always has api_data. In the user's logs, response_body often appears to be api_data itself.
2. Convert request numeric strings to int before comparisons: "1" and 1 should be treated the same.
3. Use api_event_kind + api_event_id + api_e_deck_info together for node classification; api_color_no alone is not enough.
4. Use api_midnight_flag==1 from day battle response to infer a night-battle choice prompt.
5. Use @@BattleResult for final rank/HP/result; raw battleresult response is useful for logging but less ideal for controller logic.
6. Use fleet_snapshot max_hp for taiha/chuuha calculation; deck_init_hp is battle-start HP, not max HP.
7. Treat all event-map/special-mechanic fields as extensible. Unknown numeric values should be logged, not treated as fatal errors.

Source notes:
- poi game.request/game.response events: poi docs describe request event as method/path/body and response event as method/path/body/postBody.
- poi @@BattleResult: poi Redux docs list rank, boss, mapCell, deckShipId, deckHp, deckInitHp, enemyFormation, enemyHp and recommend @@BattleResult for battle result.
- api_color_no/api_event_kind/api_rashin_* values: old/community API notes and api_start2/mapcell notes; use as hints, not as a formal stable schema.
- Current kcsapi TypeScript definitions confirm map start/next field shapes and which fields are optional, including api_eventmap, api_select_route and api_limit_state.
- poi-plugin-prophet getSpotKind supplies the contextual api_event_id + api_event_kind mappings; its internal spot kind 12 means Manual Selection and is not a raw API value.
- api_formation/engagement values: community raw API notes; combined formation 11-14 commonly maps to the four combined formations.
